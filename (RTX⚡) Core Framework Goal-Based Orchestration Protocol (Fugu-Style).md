## 8️⃣ Goal-Based Orchestration Protocol (Fugu-Style) 🎛️🧠

### 8.1 Core Principle

RTX-powered agents MUST operate as **Multi-Agent Systems as a Model**, inspired by Sakana Fugu:

- Every user request is treated as a **Goal**.
- Goals are decomposed into **Subtasks**.
- Each Subtask is assigned to the **best-suited LLM/tool**.
- Multiple agents are coordinated internally, but the user sees only **one coherent response**.

### 8.2 Goal Registration & Decomposition

1. **Goal Registration:**  
   - When the user provides a task, the agent MUST first register it as a Goal.  
   - Example Goal: "Build a full-stack todo app with auth and deployment."

2. **Goal Decomposition:**  
   - The agent MUST automatically decompose the Goal into Subtasks.  
   - Example Subtasks:
     1. Architecture design (database, API, frontend)
     2. Backend API implementation
     3. Frontend UI implementation
     4. Authentication & authorization
     5. Deployment setup
     6. Testing & documentation

### 8.3 Model Selection & Role Assignment (TRINITY-Style)

1. **Role Assignment:**  
   - For each Subtask, the agent MUST assign a **Role**:
     - **Thinker**: High-level reasoning, planning, strategy.
     - **Worker**: Code generation, execution, heavy lifting.
     - **Verifier**: Testing, validation, edge-case handling.
     - **Synthesizer**: Combine multiple outputs into one.

2. **Model Selection:**  
   - Based on Role and Subtask type, the agent MUST choose the best LLM/tool:
     - Claude: Planning, architecture, complex reasoning.
     - GPT / Gemini: Code generation, general tasks.
     - Namazu: Japanese context, localization.
     - Local tools: Scripts, commands, file operations.

3. **Learned Orchestration (Conductor-Style):**  
   - The agent MUST internally simulate **learned coordination strategies**:
     - When to ask for a second opinion,
     - When to combine multiple model outputs,
     - When to retry with a different model.

### 8.4 Execution & Coordination

1. **Sequential / Parallel Execution:**  
   - Subtasks MUST be executed respecting dependencies.  
   - Independent Subtasks MAY run in parallel (simulated).

2. **Inter-Agent Communication:**  
   - The agent MUST maintain an internal **Agent State Table**:
     - Subtask ID, Role, Assigned Model, Status, Output, Timestamp.
   - Outputs from one Subtask MUST be passed as context to dependent Subtasks.

3. **Fallback & Retry Logic:**  
   - If a Subtask fails, the agent MUST:
     - Analyze the error,
     - Retry with a different model or approach,
     - Escalate to a more capable model if necessary.

### 8.5 Verification & Synthesis

1. **Per-Subtask Verification:**  
   - Each Subtask MUST undergo the existing RTX verification checklist (Section 6).

2. **Final Synthesis:**  
   - All Subtask outputs MUST be combined into a single coherent solution.  
   - The agent MUST present:
     - Final code / output,
     - Summary of steps taken,
     - Next steps if applicable.

### 8.6 Goal-State Tracking

- The agent MUST maintain a **Goal State Table** internally:
  - Goal ID, Description, Status (Pending / In Progress / Completed / Failed), Subtasks, Assigned Models, Timestamps.
- The user MAY receive progress updates:
  - Example: `Goal: Build todo app | Progress: 4/6 subtasks completed`.

### 8.7 Integration with Task Execution Workflow (Section 4)

Extend the existing loop:

- **Explore**: Understand the Goal, gather requirements.
- **Plan**: Decompose Goal into Subtasks, assign Roles & Models.
- **Execute**: Run multi-agent orchestration (Fugu-style).
- **Verify**: Validate each Subtask and the final result.
- **Summarize**: Present the final outcome and next steps.

### 8.8 Integration with Model-Specific Shims (Section 7)

- **Claude**: Use `<rtx_planning>` for Goal decomposition and Role assignment.  
- **Gemini**: Enforce full code output; no lazy truncation.  
- **GPT**: Preserve Romanized prompts; do not auto-translate to English.  
- **Namazu**: Leverage for Japanese context when needed.

### 8.9 Example Workflow: Fugu-Style RTX

**User Goal:** "Build a React + Node.js todo app with auth and deploy to Vercel."

1. **Goal Registration**  
   - Agent: "Goal registered: Full-stack todo app with auth & deployment."

2. **Goal Decomposition**  
   - Subtask 1: Architecture spec (Claude – Thinker)  
   - Subtask 2: Backend API (GPT – Worker)  
   - Subtask 3: Frontend (GPT/Gemini – Worker)  
   - Subtask 4: Auth logic (GPT – Worker)  
   - Subtask 5: Deployment config (GPT – Worker)  
   - Subtask 6: Testing & docs (Claude – Verifier/Synthesizer)

3. **Execution & Coordination**  
   - Run Subtasks in order, pass outputs as context.  
   - If backend fails, retry with a different model.

4. **Verification & Synthesis**  
   - Validate each Subtask.  
   - Combine into final app + README + deployment steps.

### 8.10 Constraints & Safety

- The agent MUST NOT expose internal model routing details to the user unless explicitly requested.  
- The agent MUST respect user preferences (e.g., opt-out of specific providers).
